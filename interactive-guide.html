Smart Elephant is a repository that uses an approval-first AI delivery workflow to keep your work auditable, reviewable, and reproducible. This interactive guide will help you understand how to use it, even if you're not a technical person!

What is Smart Elephant?

Smart Elephant is a system that helps teams build software with AI assistance while maintaining quality and control. Think of it as having an AI assistant that needs your approval before making changes.

üîë Key Principles

1 Plan Before Work

Every significant change starts with a plan. The AI creates a plan document that describes what it will do, and a human must approve it before any work begins.

Example: Before adding a new feature, the AI writes a plan explaining the scope, deliverables, and timeline. You review and approve it.
2 Everything is Auditable

Every prompt (instruction) given to the AI and every file it generates is logged. This creates a complete history of who did what and why.

## Example log entry in prompts.md: Date: 2025-10-24 Prompt: "Create user authentication system" Files Generated: - auth/login.js - auth/register.js - auth/middleware.js
3 Small, Testable Changes

Changes are broken into small pieces that can be tested independently. This makes it easier to find and fix problems.

üí° Tip: Instead of one huge change, make several small changes. Each one should do something specific and testable.
4 Specialized Chat Modes

Different phases of development use different "chat modes" - specialized AI assistants that know exactly what to do in each phase.

Phases:
  • Setup: Initialize your project
  • Inception: Create user stories and architecture
  • Construction: Build the actual code and infrastructure

üí¨ Understanding Chat Modes

Chat modes are specialized AI assistants for different phases of development. Think of them as experts in specific tasks.

Setup Phase

Setup

1. Setup Chat Mode

Initializes the AI delivery workflow and repository structure. Use this first when starting any project.

Creates: Project folders, documentation templates, and workflow files

Inception Phase

Inception

2.1 Inception - Stories

Generates user stories from product requirements. Helps translate ideas into actionable development tasks.

// Example usage: Input: "Build an online bookstore" Output: - User Story: Browse books by category - User Story: Add books to shopping cart - User Story: Checkout and payment - User Story: Order history
Inception

2.2 Inception - Units

Implements software units from component designs. Turns design documents into actual code.

Input: Component design specification
Output: Implementation files (services, models, utilities)
Inception

2.3 Inception - Architecture

Designs architecture and deployment planning. Decides how different parts of the system work together.

Creates: Architecture diagrams, deployment plans, cloud resource specifications

Construction Phase

Construction

3.1 Construction - Domain

Domain-driven component model design. Creates the "blueprint" of your system's components.

üí° What's a component model? It's like a map showing all the pieces of your system and how they connect.
Construction

3.2 Construction - Code

Code generation and deployment automation. Creates the infrastructure code needed to deploy your application.

Generates: Terraform files, deployment scripts, configuration files

Builds REST APIs and Infrastructure as Code. Creates the actual API endpoints your application will use.

üìñ Real-World Examples

üå± Greenfield Example: New E-Commerce Platform

Scenario: You want to build a brand new online store from scratch.

Process:
  1. Start with product vision: "Online bookstore with recommendations"
  2. Generate user stories: browsing, cart, checkout, reviews
  3. Design system architecture: web frontend, API backend, database
  4. Create all components: user service, product service, order service
  5. Build APIs: /products, /cart, /orders, /reviews
  6. Generate deployment: cloud infrastructure, database, storage
  7. Deploy everything as a complete new system
üèóÔ∏è Brownfield Example: Add AI Search to Existing Store

Scenario: You have an existing bookstore, and you want to add AI-powered search.

Process:
  1. Analyze existing search implementation
  2. Create enhancement stories: "As a user, I want smart search suggestions"
  3. Update domain models to include AI search component
  4. Design integration with existing search
  5. Implement AI search service alongside old search
  6. Create new API endpoint: /api/v2/ai-search
  7. Keep old endpoint: /api/v1/search (for compatibility)
  8. Update infrastructure to add ML model hosting
  9. Deploy with blue-green strategy (can rollback if issues)
‚ö†Ô∏è Key Difference: Must ensure old functionality keeps working while adding new features!
